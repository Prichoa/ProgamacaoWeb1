Como o Nexus Game Hub foi construído

Aqui você encontra a estrutura de cada jogo e os motivos técnicos que orientaram as escolhas de Kelvyn, Erika e Lucas. O conteúdo foi pensado para responder perguntas no nível de um desenvolvedor sênior em front-end.

O que é front-end?

Front-end é tudo que você vê e clica no navegador. É como a "cara" de um site ou aplicativo. O HTML é a estrutura (o esqueleto), o CSS é a beleza (roupa e maquiagem), e o JavaScript é a lógica (o cérebro que faz as coisas funcionarem).

🏗️ HTML

Linguagem de marcação. Define os elementos: botões, títulos, placar, histórico. É como escrever "aqui tem um botão" e "ali tem um texto".

🎨 CSS

Folha de estilos. Define cores, tamanhos, posições, animações. Transforma um HTML simples em algo bonito e responsivo (que funciona em qualquer tela).

⚙️ JavaScript

Linguagem de programação. Faz o interativo: quando você clica, o jogo reage. Calcula resultados, atualiza números, guarda dados.

Perguntas de front-end sênior

1. Por que você usou `localStorage` para armazenar o placar?

O básico: localStorage é um espaço de armazenamento que fica no navegador do usuário, permitindo guardar informações mesmo depois que ele sai do site. No meu projeto: Usei para guardar quantas vitórias, empates e derrotas o jogador teve em cada partida. Quando ele volta ao jogo, o placar não zera. Por quê? Isso mantém a experiência contínua, sem precisar de um servidor que custe dinheiro. Se usássemos um servidor, teríamos que enviar dados toda hora e esperar uma resposta, deixando o jogo mais lento.

2. Como você separou a animação de rolagem do estado do jogo?

O básico: Animação é quando algo se move ou muda na tela. Estado é a informação real que você quer guardar (quem venceu, quantas vitórias). No meu projeto: Quando o jogador aperta "Lançar", os dados giram e fazem barulho na tela por alguns segundos (animação). Depois que terminam, meu JavaScript calcula quem ganhou e atualiza o placar (estado). Por quê separar? Assim, a tela fica suave e o jogo não trava enquanto os dados estão girando. Se eu misturasse os dois, o navegador ficaria travado durante a animação.

3. Quais critérios você usou para o layout e a hierarquia visual?

O básico: Hierarquia visual significa organizar o que é mais importante para aparecer em destaque. Contraste significa usar cores ou tamanhos bem diferentes. No meu projeto: O placar aparece no topo grande e colorido. Os dados ficam no meio bem visíveis. O resultado pisca em uma faixa chamativa no meio. Por quê? Quando alguém abre a página, os olhos vão direto para o que importa: quanto ganhou até agora, que números saíram e se venceu. O histórico de rodas antigas fica em baixo, porque é menos importante que o jogo atual.

Perguntas de front-end sênior

1. Por que você adicionou a barra de streak?

O básico: Streak significa uma sequência de sucessos seguidos. Na página tem um contador que mostra "Sequência: 0". No meu projeto: Cada vez que o jogador acerta, o número sobe (sequência de 1, 2, 3...). Se errar, volta para zero. Por quê? Psicologicamente, ver uma sequência crescente motiva o usuário a continuar jogando para não "quebrar" a sequência. É como jogar videogame e querer chegar ao próximo nível. Sem isso, o jogo seria só "você venceu" ou "você perdeu", sem um objetivo extra.

2. Como você fez a interação de seleção de cara/coroa segura e acessível?

O básico: Um botão `button` é um elemento HTML que qualquer um pode clicar. Acessível significa que funciona para qualquer pessoa, até quem não consegue usar mouse (usa teclado). No meu projeto: Em vez de usar divs e simulador de botão, usei elementos reais `<button>`. Quando alguém clica, o botão fica com a classe `selected` para ficar destacado. Por quê? Elementos `button` reais funcionam com teclado automaticamente. Pessoa cega ou com deficiência motora consegue jogar usando Tab e Enter. Se eu fizesse só com divs, não funcionaria para eles.

3. Qual foi sua decisão de performance ao animar a moeda?

O básico: Performance é velocidade. Quando você anima algo, o navegador precisa redesenhar a tela muitas vezes. Há duas formas: mudar o HTML a cada frame (lento) ou mudar apenas o CSS (rápido). No meu projeto: A moeda gira usando uma classe CSS chamada `flipping` que tem uma animação. Meu JavaScript só muda a classe uma vez, e o CSS cuida de toda a rotação. Por quê? O CSS é otimizado para animações e deixa a moeda girar suave. Se eu mudasse o HTML ou os estilos com JavaScript a cada frame, o navegador ficaria lento e a animação trepidaria. Além disso, em celulares com bateria fraca, é fácil desligar a animação CSS.

Perguntas de front-end sênior

1. Por que organizar as regras em um objeto `CHOICES`?

O básico: Um objeto é uma forma de guardar informações relacionadas juntas. Por exemplo, guardar qual é o ícone do pedra, qual é o nome, e qual opção ela vence. No meu projeto: Ao invés de ter variáveis espalhadas como `pedra_icon`, `pedra_nome`, `pedra_bate`, uso um objeto `CHOICES` que tem tudo junto: `{ rock: { icon: '✊', label: 'PEDRA', beats: 'scissors' } }`. Por quê? Se eu quiser adicionar uma 4ª opção (tipo "Pedra de Fogo"), basta adicionar ela no objeto, e toda a lógica funciona igual. Sem o objeto, eu teria que mudar o código em vários lugares. Com o objeto, é só adicionar uma linha.

2. Como você lidou com feedback visual no resultado?

O básico: Feedback visual significa mostrar na tela o que aconteceu. Classe CSS é um nome que você coloca no HTML para aplicar estilos do CSS. No meu projeto: Quando o jogador vence, a página adiciona uma classe chamada `result-win` que deixa tudo verde e alegre. Quando perde, adiciona `result-lose` que deixa tudo vermelho. Quando empata, adiciona `result-draw` que deixa azul. Por quê? Assim meu JavaScript fica simples, apenas diz "vitória" ou "derrota". O CSS cuida de deixar bonito (cores, brilhos, animações). Se eu colocasse todas as cores e animações no JavaScript, o código fica bagunçado e difícil de entender.

3. Quais práticas você seguiu para manter o código reutilizável?

O básico: Código reutilizável significa escrever uma função uma vez e usar em vários lugares, ao invés de copiar a mesma coisa várias vezes. No meu projeto: Criei funções como `bumpNum()` (anima um número no placar), `renderHistory()` (desenha o histórico), `updateHUD()` (atualiza a tela). Se eu precisar mudar como o histórico é desenhado, mudo em um único lugar. Por quê? Se eu copiasse o código do histórico 3 vezes, teria que mudar 3 vezes. Com uma função, mudo uma vez e pronto. Além disso, isso deixa fácil para alguém escrever testes automatizados que verificam se o jogo está funcionando certo.

Visão geral técnica

O projeto segue uma arquitetura de front-end modular: cada jogo tem sua própria pasta com HTML, CSS e JS. A página inicial fica em `Main/index.html` e faz a ponte entre as arenas. A nova página “Sobre” explica o papel de cada desenvolvedor e os motivos de cada escolha técnica.

  • Pasta: cada jogo mora em sua própria pasta (Dados, caraoucoroa, jokenpo) para organizar melhor os arquivos.
  • HTML: estrutura da página. Define os botões, placar, histórico e tudo que você vê na tela.
  • CSS: beleza e aparência. Deixa as coisas coloridas, com animações e responsivas (funciona em celular, tablet, computador).
  • JavaScript: a lógica do jogo. Calcula quem venceu, atualiza o placar, guarda no `localStorage`.
  • localStorage: um lugar seguro no navegador para guardar dados do usuário, mesmo depois que ele sai do site.
  • Tema compartilhado: a página `gameshared.css` tem estilos que todos os jogos usam (fontes, cores, placar padrão), evitando repetir código.
